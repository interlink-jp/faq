# AI メール返信下書き機能 — 開発指示書

## 1. 本書の位置づけ

本書は、現在開発中のメール共有管理システムに「AI による返信下書き機能」を追加するための設計・実装指示書である。開発担当者および開発に使用する Claude / Claude Code に対し、実装方針・アーキテクチャ・技術選定の根拠を共有することを目的とする。

---

## 2. 機能概要

### 2.1 目的

問い合わせメールへの返信業務を効率化する。AI が過去の返信履歴と FAQ を参照し、返信文の下書き（原案）を自動生成する。担当者は下書きを確認・編集した上で送信する。

### 2.2 基本方針

- ローカル LLM は使用しない。すべてクラウド API で構成する。
- 返信文の生成には Claude Sonnet API を使用する。
- テキストのベクトル化（Embedding）には外部の Embedding API を使用する（Claude は Embedding API を提供していないため）。

---

## 3. ユーザー体験（UX）

### 3.1 トリガー方式: ボタン押下型

メール受信時に自動で下書きを生成する方式は採用しない。担当者がメール詳細画面で「AI 原案作成」ボタンを押したときにのみ生成する。

**ボタン押下型を選択する理由:**

- 必要な時だけ API を呼ぶため、コストが最小限になる
- 返信不要なメール（お礼、スパム等）に無駄な生成が発生しない
- ボタン押下時点の最新の履歴・FAQ を参照できる（受信時生成だと時間差で情報が古くなる）
- 担当者が「AI に任せるか自分で書くか」を判断できる主体性が残る
- 「AI が作った原案を自分が確認・修正して送る」というワークフローが自然に成立する

### 3.2 画面構成

担当者が「AI 原案作成」ボタンを押すと、以下の2つが表示される。

#### (A) 返信下書きエリア

- Claude が生成した返信文を表示する
- 担当者はこの文面を直接編集できる
- 編集後、そのまま返信として送信できる

#### (B) 参考情報パネル

AI が下書きを作成する際に参照した情報を一覧表示する。これはお客様には見えず、担当者のみが確認できる情報である。

表示項目:

- 参照した過去メール（件名、日時、類似度スコア）→ クリックで全文表示
- 参照した FAQ 項目（タイトル、類似度スコア）→ クリックで表示
- Claude が実際に根拠とした情報の番号（Claude の出力に含めさせる）
- メンバー ID リンク（後述）

#### メンバー ID の自動検出とリンク表示

問い合わせメールの本文中に数字 10 桁のメンバー ID が含まれている場合、参考情報パネルに該当会員の管理画面へのリンクを自動表示する。

- 検出条件: メール本文から正規表現等で 10 桁の数字列を抽出する
- リンク先 URL: `https://luna/home/ILmanAdmin/M1/M10003.asp?CUST_ID={メンバーID}`
- 例: メール本文に「1234567890」が含まれている場合、参考情報パネルに「会員情報: 1234567890」としてリンクを表示し、クリックすると `https://luna/home/ILmanAdmin/M1/M10003.asp?CUST_ID=1234567890` を開く
- 複数のメンバー ID が検出された場合は、すべてをリンクとして表示する
- この機能は AI（Claude API）を経由しない。メール本文のパースのみで完結するため、追加の API コストは発生しない

**参考情報パネルが重要な理由:**

- 担当者が下書きの妥当性を判断できる
- 古い情報（料金改定前の FAQ 等）を根拠にしている場合に気づける
- 追加の API コストはほぼゼロ（検索結果を画面に表示するだけ）

## 4. システムアーキテクチャ

### 4.1 全体構成

```
┌─────────────────────────────────────────────────────────┐
│  事前準備（初回 + 新規メール到着時に随時追加）            │
│                                                         │
│  過去メール ──→ Embedding API ──→ ベクトル DB に保存     │
│  FAQ データ ──→ Embedding API ──→ ベクトル DB に保存     │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  返信時（「AI 原案作成」ボタン押下時）                    │
│                                                         │
│  1. 問い合わせメールの内容を Embedding API でベクトル化   │
│  2. ベクトル DB で類似検索                                │
│     → 類似の過去メール+返信ペアを 2〜3 件取得            │
│     → 関連する FAQ を数件取得                            │
│  3. 以下をまとめて Claude Sonnet API に送信               │
│     - システムプロンプト（返信ルール・トーン等）          │
│     - 当該メールのスレッド全体                           │
│     - 検索で取得した過去メール+返信ペア                  │
│     - 検索で取得した FAQ                                 │
│  4. Claude の出力（返信文 + 参照情報番号）を画面に表示    │
└─────────────────────────────────────────────────────────┘
```

### 4.2 重要な設計ポイント: 過去メール全件を毎回読ませるわけではない

過去メールが 1 万通あっても 10 万通あっても、Claude に渡すトークン数は変わらない。ベクトル検索で「意味的に近い」過去メールを数件だけ取得し、それだけを Claude に渡す。

ベクトル検索は意味で検索するため、「料金を知りたい」「価格について」「いくらですか」といった表現の揺れがあっても正しくヒットする。

## 5. 技術選定

### 5.1 返信文生成: Claude Sonnet API

- モデル: `claude-sonnet-4-6`（または利用時点の最新の Sonnet）
- 用途: 返信文の下書き生成
- 料金: 入力 $3 / 100 万トークン、出力 $15 / 100 万トークン
- 1 通あたりの推定コスト: 約 3〜10 円

### 5.2 Embedding（テキストのベクトル化）

Claude は Embedding API を提供していないため、以下のいずれかを使用する。後から差し替え可能なように、Embedding 処理は抽象化（インターフェース分離）して実装すること。

**候補（いずれか 1 つを選択）:**

| サービス | 料金目安 | 備考 |
|---|---|---|
| Voyage AI | $0.02 / 100 万トークン | Anthropic が推奨。日本語精度良好 |
| OpenAI text-embedding-3-small | $0.02 / 100 万トークン | 実績豊富。日本語対応 |
| Google text-embedding-004 | 無料枠あり | Google Cloud 利用時に便利 |

いずれを選んでも、Embedding のコストは 1 回の検索あたり 0.01 円以下であり、コスト面での差はほぼない。選定基準は主に日本語の検索精度。

**今回の選定: Voyage AI を採用する。** Anthropic が推奨するプロバイダーであり、日本語の検索精度が良好なため。ただし、将来的にプロバイダーを切り替える可能性を考慮し、Embedding 処理はインターフェースを分離して実装すること（上述の通り）。

### 5.3 ベクトル DB

以下のいずれかを選択する。

| サービス | 特徴 |
|---|---|
| pgvector（PostgreSQL 拡張） | 既存の PostgreSQL に追加できる。小〜中規模向き |
| Pinecone | マネージドサービス。運用負荷が低い |
| Qdrant | オープンソース。セルフホスト可能 |

既存システムが PostgreSQL を使用している場合は、pgvector が最もシンプルな選択肢。新たなインフラが不要。

#### ベクトル DB の更新スケジュール

ベクトル DB のデータ（過去メール・FAQ のベクトル）は、以下のスケジュールで更新する。

- **定期更新**: 毎週土曜日 13:00 に自動実行する。新規に蓄積されたメールのベクトル化および FAQ の再ベクトル化を行う
- **臨時更新**: 管理者が管理画面から任意の日時を指定して手動実行できるようにする。FAQ の大幅な更新や、過去メールの一括インポート時などに使用する
- **定期更新の設定変更**: 定期更新の有無（ON / OFF）および実行日時は、管理者が管理画面から変更できるものとする

## 6. Claude に渡すプロンプトの設計

### 6.1 システムプロンプト（毎回共通）

```
あなたはインターネットサービスプロバイダの問い合わせ対応担当です。
お客様からの問い合わせメールに対する返信の下書きを作成してください。

## ルール
- 丁寧で正確な日本語のビジネスメールを作成すること
- 不確かな情報は含めないこと
- 参照した情報（過去メールや FAQ）の番号を返信文の後に明記すること
- 回答できない内容は「確認して改めてご連絡いたします」と記載すること
- 返信文のみを出力すること（説明や前置きは不要）

## 出力形式
まず返信文を出力し、その後に以下の形式で参照情報を記載してください:

---
参照情報: [MAIL-1], [FAQ-2]
---
```

### 6.2 ユーザープロンプト（メールごとに動的に組み立て）

```
## 当該メールスレッド
[ここにスレッド全文を挿入]

## 参考: 類似の過去メールと返信
[MAIL-1]
問い合わせ: [過去の問い合わせ文]
返信: [過去の返信文]

[MAIL-2]
問い合わせ: [過去の問い合わせ文]
返信: [過去の返信文]

## 参考: 関連 FAQ
[FAQ-1]
Q: [質問]
A: [回答]

[FAQ-2]
Q: [質問]
A: [回答]

上記を参考に、最新の問い合わせメールに対する返信の下書きを作成してください。
```

## 7. 事前作業（実装開始前に人間が行う）

以下は Claude Code では実施できないため、担当者が事前に済ませておくこと。

### ① Anthropic API アカウント作成
- [console.anthropic.com](https://console.anthropic.com) でアカウント作成
- クレジットカード登録・クレジット入金
- API キーを発行してコピーしておく

### ② Voyage AI アカウント作成
- [dash.voyageai.com](https://dash.voyageai.com) でアカウント作成
- API キーを発行してコピーしておく

### ③ 環境変数の設定
- `.env` ファイルに以下を追記する（Claude Code が記載箇所を指示する）
ANTHROPIC_API_KEY=（①で取得したキー）
VOYAGE_API_KEY=（②で取得したキー）
- デプロイ先（Railway / Render 等）の管理画面にも同じキーを登録する

### ④ 既存メールデータの準備
- 旧システムからメールデータをCSVで全件エクスポート
- エクスポートしたファイルのパスを控えておく（Claude Code 実行時に伝える）

## 8. 実装手順

### フェーズ 1: Embedding 基盤の構築

1. Embedding API クライアントを作成する
   - プロバイダーを後から差し替えられるようにインターフェースを定義する
   - 入力: テキスト文字列 → 出力: ベクトル（数値配列）
2. ベクトル DB をセットアップする
   - pgvector を使う場合: 既存 DB にエクステンションを追加し、ベクトル用テーブルを作成
3. 過去メール・FAQ のベクトル化バッチを作成する
   - 既存の過去メール（問い合わせ+返信ペア）を一括でベクトル化して保存
   - FAQ データを一括でベクトル化して保存
4. 新規メール到着時にベクトルを自動追加する仕組みを作る

### フェーズ 2: 検索機能の実装

5. 類似メール検索 API を実装する
   - 入力: 問い合わせメールのテキスト
   - 処理: Embedding → ベクトル DB で類似検索
   - 出力: 類似度上位 2〜3 件の過去メール+返信ペア（類似度スコア付き）
6. 関連 FAQ 検索 API を実装する
   - 同様の仕組みで FAQ から関連項目を数件取得

### フェーズ 3: AI 下書き生成機能の実装

7. Claude API クライアントを作成する
   - 6 章のプロンプト設計に基づき、コンテキストを組み立てて Claude API を呼ぶ
   - レスポンスから返信文と参照情報番号を分離してパースする
8. 「AI 原案作成」ボタンと UI を実装する
   - メール詳細画面にボタンを追加
   - ボタン押下で検索 → Claude API 呼び出し → 結果表示の一連のフローを実装
   - 返信下書きエリアと参考情報パネルの 2 エリア構成で表示する

### フェーズ 4: 改善

9. プロンプトキャッシュの導入
   - システムプロンプトや FAQ など毎回同じ部分をキャッシュして入力コストを削減
10. 利用ログの蓄積と分析
    - どのメールで AI 原案が使われたか、編集量はどの程度だったかを記録
    - プロンプト改善のフィードバックループを構築

## 9. コスト見積もり

### 1 通あたりのコスト（Claude Sonnet）

| ケース | 入力トークン | 出力トークン | コスト |
|---|---|---|---|
| 軽い問い合わせ | ~5,000 | ~400 | 約 3 円 |
| 普通の問い合わせ | ~10,000 | ~500 | 約 6 円 |
| 複雑な問い合わせ | ~20,000 | ~800 | 約 10 円 |

### 月間コスト試算（1 日 30 通利用の場合）

- Claude API: 月 900 通 × 6 円 ≒ 約 5,400 円
- Embedding API: 月数十円（誤差レベル）
- ベクトル DB: pgvector なら追加コストなし（既存 DB 利用）
- **合計: 月額 約 6,000 円程度**

---

## 10. 注意事項

- Claude API はデータをモデルの学習に使用しないポリシーだが、顧客メールを外部 API に送信することに対し、社内のセキュリティポリシーやプライバシーポリシーとの整合性を事前に確認すること
- FAQ の内容が更新された場合は、次回の定期更新（毎週土曜 13:00）を待つか、管理者が臨時更新を実行してベクトル DB に反映すること
- AI が生成した下書きは必ず担当者が確認してから送信する運用とし、自動送信は行わないこと
